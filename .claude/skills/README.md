## Updating a skill

When you want to refresh a skill from its upstream source:

1. Click the skill name in the table to open the source repository.
2. Clone (or pull) the repository locally.
3. Replace the contents of `.claude/skills/<skill-name>/` with the upstream `<source-repo>/skills/<skill-name>/` contents.
4. Reload the Claude Code session so the new metadata is picked up.

After updating, **manually refresh the `Version` and `Last edit` columns**. The `Last edit` value is the date of the latest commit in the upstream repository that touched the skill's folder — query it with:

```
gh api 'repos/<owner>/<repo>/commits?path=<source-path>&per_page=1' --jq '.[0].commit.author.date'
```

Replace `<source-path>` with the path inside the repo (typically `skills/<skill-name>` for skills bundled in a `skills/` folder, or the file path for single-file sources like `commit`). Convert the returned ISO date to `dd/mm/yyyy` for the table.

## Column meaning

- **Skill** — the folder name under `.claude/skills/`, which is also the name Claude Code uses when invoking the skill. The link points to the upstream source on GitHub.
- **Version** — the version declared by the upstream skill. Most skills declare it in `SKILL.md` frontmatter under `metadata.version`; another place to check is `tile.json` under `"version"`. Em dash (—) means the upstream skill doesn't declare a version (e.g., the `commit` skill, which is a plain markdown blob without versioning).
- **Last edit** — the date of the latest commit in the **upstream source repository** that touched the skill's source path (not when the local copy was last modified). Use this to know when the upstream got an update — if your local copy was installed before this date, you're behind. Format is `dd/mm/yyyy`.
- **Description** — the `description` from the skill's frontmatter, which is what Claude Code reads to decide when to auto-invoke the skill.
